The graphene window system starts up initially without a
.graphene-geometry file, and the default geometry is so big that the
initial emacs frame goes off the display. You can't get it back using
the mouse. You have to use Notepad to edit the .graphene-geometry file
to a reasonable size and restart emacs. I put in a new customize
variable `graphene-initial-geometry` and set it to the same values
that a bare-naked emacs uses on startup.

I probably should drop the customize variable and just put the new
value into `graphene-load-frame-geometry`.

That solves the initial startup problem. The problem re-occurs if the
user exits emacs when its frame is maximized. For some reason, the
difference between the edges of the frame and the edges of the display
computed by emacs are wrong when the frame is maximized. See
`graphene-get-margins-pixels`. If emacs/graphene reads out the values
in the .graphene-geometry file when emacs next starts up, it doesn't
set the frame to maximize, and the wrong geometry values push the
bottom right conrner of the frame off-display. You get the same
problem as initial startup.

The best solution, I think, is to write out the maximize state of the
frame when emacs is closed, and read it back in the next time it is
started up, then set the new frame to the same state. Trouble is,
emacs maximize frame functions don't work across different OSes
(Windows, Linux, OSX,...). Maybe someone has fixed it...

Today emacs got in this wierd state where

    (frame-parameter nil 'fullscreen)

returned `fullscreen`. Not `nil` or `maximized` as it should, but `fullscreen`. Looking inside the alist returned by `(frame-parameters)` showed that, sure enough, there was a pair `(fullscreen . fullscreen)` in there. At the same time, the emacs frame covered the entire display, cincluding the Windows toolbar at the bottom of the display. Maybe it's supposed to do that sometimes? I didn't do it on purpose.

This is an exploration of a rather heavy-weight way to solve the
problem: upon closing emacs, check for frame geometry values that are
too big, then fix them. Probably I should just do it right. Given what
I see below, it's most likely easier anyway.

if the display is maximized, the given frame size is fine, except that emacs won't start up again in maximized mode, and the frame will overlap the edges of the display
solutions
1. record the state of the frame (full-screen) and set the state of the frame when emacs starts up again.
2. treat the recorded geometry the same as any other, and fit to the display on startup

record the frame (height,width) characters of the existing frame

if (top, left) is off-display, increment by char (height,width) pixels until it is back on-display, record delta pixels

if (bottom,right) is off-display, decrement by char (height,width) pixels until it is back on-display, record delta pixels
